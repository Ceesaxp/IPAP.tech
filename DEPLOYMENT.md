# Deployment — one-time setup

The build pipeline is fully wired in `.github/workflows/deploy.yml`. To make `ipap.tech` actually serve from GitHub Pages, complete these steps on the GitHub side. **No secrets or credentials need to live in this repo** — Pages deploys with the built-in `GITHUB_TOKEN`.

## 1. Repository settings (one-time)

**Settings → Pages**

- **Source:** `GitHub Actions` (not "Deploy from branch")
- **Custom domain:** `ipap.tech`
- **Enforce HTTPS:** checked (after DNS propagates)

The `static/CNAME` file in this repo writes `ipap.tech` into the build output, so the custom domain stays set even when the workflow republishes.

## 2. DNS (at your registrar)

Point `ipap.tech` at GitHub Pages. The simplest correct configuration:

| Type    | Host   | Value                  |
| ------- | ------ | ---------------------- |
| `A`     | `@`    | `185.199.108.153`      |
| `A`     | `@`    | `185.199.109.153`      |
| `A`     | `@`    | `185.199.110.153`      |
| `A`     | `@`    | `185.199.111.153`      |
| `AAAA`  | `@`    | `2606:50c0:8000::153`  |
| `AAAA`  | `@`    | `2606:50c0:8001::153`  |
| `AAAA`  | `@`    | `2606:50c0:8002::153`  |
| `AAAA`  | `@`    | `2606:50c0:8003::153`  |
| `CNAME` | `www`  | `<gh-user>.github.io.` |

(GitHub Pages IPs as of late 2024 — verify against <https://docs.github.com/en/pages/configuring-a-custom-domain-for-your-github-pages-site>.)

DNS can take up to an hour to fully propagate; HTTPS provisioning takes another 5–15 minutes after that.

## 3. Workflow permissions (one-time)

**Settings → Actions → General → Workflow permissions**

- "Read and write permissions" should be selected.
- Pages-specific permissions (`pages: write`, `id-token: write`) are already declared in the workflow file.

## 4. Optional but recommended

- **Branch protection** on `main`: require the `PR build check` workflow to pass before merge.
- **Environment** named `github-pages` (auto-created on first deploy). Restrict deploys to the `main` branch.

## 6. Migration from the old EC2 host

The previous deployment used `make deploy` / `deploy.sh` to rsync to `ec2-user@rsync-aws.fft`. Those files have been removed. When you're ready to retire the EC2 host:

1. Confirm `ipap.tech` is serving from GitHub Pages with HTTPS (green padlock).
2. Tear down the EC2 webroot (`/var/www/html/ipap.tech/htdocs/`) — or leave it running until you're confident in the new setup.
3. If the old host has any inbound links (CDN, redirects), update them to point at GitHub Pages.
